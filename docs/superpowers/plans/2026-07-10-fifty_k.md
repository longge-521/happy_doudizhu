# 510K 可靠结算与专属 AI 优化实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 修正 510K 压制顺序、5/10/K 计分、局中与终局欢乐豆结算、分布式卡局问题，并为机器人、提示和托管提供同一套 510K 专属策略 AI。

**Architecture:** 领域层只负责玩法规则和确定性结算数据；应用层统一编排真人、AI、托管和超时动作；数据库分别对牌墩与终局做幂等结算；WebSocket 只广播已成功持久化的实际变化。经典模式继续使用 DouZero，510K 先使用合法动作生成、局面评分和有限蒙特卡洛采样，后续再单独训练自博弈模型。

**Tech Stack:** Python 3.10、FastAPI、SQLAlchemy、Redis、RabbitMQ、Pytest、Vue 3、Pinia、TypeScript、Vitest、PyTorch（仅复用现有运行环境）。

## Global Constraints

- 510K 压制顺序固定为 `王炸 > 真510K > 普通炸弹 > 假510K > 普通牌型`。
- 分牌固定为 `5=5分、10=10分、K=20分`，整副牌总分 140。
- 局中金额为 `牌墩分 × 牌局底分 × 牌墩结束时倍数`。
- 每名输家终局先扣 `4500 × 最终倍数`，再扣 `剩余手牌分 × 牌局底分 × 最终倍数`。
- 数据库余额不得为负；赢家只获得真人实际扣款，机器人理论扣款由系统虚拟账户承担。
- 修复前先写能复现问题的失败测试；开发阶段只运行相关定向测试，最终再执行全量回归。
- 普通单元测试必须覆盖 `DISTRIBUTED_MODE=False` 或使用内存适配器，不连接真实 Redis/RabbitMQ。
- 不改无关代码，不批量格式化，不覆盖当前工作区已有修改。
- 开发完成后更新根目录 `README.md`；用户人工确认前不得执行 `git commit`。
