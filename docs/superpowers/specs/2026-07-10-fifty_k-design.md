# 510K 各自为战玩法可靠结算与专属 AI 设计规格

## 1. 目标

本规格用于修正现有 510K 玩法的牌型压制、分牌计分、欢乐豆可靠结算、终局汇总和 AI 决策问题。

玩法保持三人各自为战：使用完整 54 张牌，每人 18 张，无底牌、无地主阵营，持有黑桃 3 的玩家获得首出权，任意玩家首先出完手牌即获胜。

本次设计必须同时满足：

- 510K 规则只在 `play_mode == "fifty_k"` 时生效，不影响经典和不洗牌模式。
- 局中每个含分牌的牌墩都实时结算欢乐豆，重复命令不得重复扣款。
- 终局固定输赢、剩余手牌分和整局累计变化可核对、可重放、可恢复。
- 单机与 Redis/RabbitMQ 分布式链路产生相同的领域结果和 WebSocket 事件。
- 机器人、提示和托管共用同一套 510K 候选动作排序。

## 2. 玩法规则

### 2.1 特殊牌型与压制顺序

- 真 510K：同一花色的 5、10、K。
- 假 510K：花色不完全相同的 5、10、K。
- 最终压制顺序固定为：

  `王炸 > 真510K > 普通炸弹 > 假510K > 普通牌型`

- 普通炸弹之间按主点数比较。
- 两手真 510K 视为同级，互相不能压制。
- 两手假 510K 视为同级，互相不能压制。
- 真/假 510K 的识别与压制必须显式接收玩法模式；经典模式中的 5、10、K 组合仍按原斗地主规则处理。

### 2.2 分牌定义

| 分牌 | 分值 |
| --- | ---: |
| 5 | 5 分 |
| 10 | 10 分 |
| K | 20 分 |

一副牌共有 4 张 5、4 张 10、4 张 K，整局分牌总分为：

`4 × (5 + 10 + 20) = 140 分`

### 2.3 牌墩吃分

一次牌墩从某位玩家首出开始，直到另外两名玩家连续过牌结束。最后一次有效出牌的玩家为牌墩赢家，收取本牌墩中三家已经打出的全部分牌。

设：

- `S`：本牌墩 5、10、K 的分值总和。
- `B`：牌局底分。
- `M_trick`：牌墩结算发生时的当前倍数。

本牌墩欢乐豆池为：

`P_trick = S × B × M_trick`

当 `S > 0` 时：

- 牌墩赢家增加 `P_trick` 欢乐豆。
- 另外两名玩家合计扣除 `P_trick`，偶数时各扣一半；奇数时按稳定座位顺序分别扣除向下取整和向上取整值。
- 牌墩赢家的局内累计分增加 `S`。
- `S == 0` 时只结束牌墩，不生成欢乐豆流水。

示例：牌墩包含一张 5、一张 10 和一张 K，`S = 35`；底分 `B = 80`，当前倍数 `M_trick = 2`，则欢乐豆池为 `35 × 80 × 2 = 5600`。赢家 `+5600`，另外两家各 `-2800`。

### 2.4 倍数口径

- 牌墩结算使用该牌墩结束瞬间的当前倍数 `M_trick`，已经完成的牌墩不因后续倍数变化而追溯重算。
- 终局结算使用对局结束瞬间的最终倍数 `M_final`。
- 倍数来源沿用房间统一的 `room.multiplier`；普通炸弹、王炸等既有翻倍行为只更新这一处事实来源。
- 文档和协议不得分别维护第二套倍数，避免客户端与服务端计算不一致。

### 2.5 最后一手与终局结算

玩家出完最后一手时按以下顺序结算：

1. 将当前尚未结束的牌墩判给出完牌的赢家；若牌墩含分牌，先按 2.3 执行一次牌墩结算。
2. 两名输家分别扣除固定失败欢乐豆：`4500 × M_final`。
3. 计算每名输家剩余手牌中的分牌总分 `R_i`，再分别扣除：`R_i × B × M_final`。
4. 赢家获得两名输家的固定失败扣款与剩余分牌扣款总和，保证终局流水零和。
5. 局内得分只记录实际吃到或终局收割到的分牌；不得从输家已经吃到的历史得分中再次扣除剩余手牌分。

每名输家的理论终局变化为：

`D_i = -(4500 × M_final + R_i × B × M_final)`

赢家理论终局变化为：

`D_winner = -(D_loser_1 + D_loser_2)`

示例：底分 80、最终倍数 2；输家 A 剩余 5 和 K，`R_A = 25`；输家 B 剩余 10，`R_B = 10`：

- A：`-(4500×2 + 25×80×2) = -13000`
- B：`-(4500×2 + 10×80×2) = -10600`
- 赢家：`+23600`

结算弹窗显示的“总变化”必须是：

`整局所有牌墩欢乐豆变化 + 终局固定输赢变化 + 剩余手牌分变化`

### 2.6 余额不足

- 数据库余额不得出现负数。
- 每笔结算先锁定真人玩家档案，实际扣款为 `min(应扣欢乐豆, 当前余额)`。
- 赢家只获得真人输家实际扣除的金额；机器人对应的理论扣款由系统虚拟账户承担。
- WebSocket 和终局弹窗同时展示实际变化，不使用理论变化伪造账户余额。

## 3. 可靠结算架构

### 3.1 房间状态

Redis 房间快照必须序列化并恢复：

- `scores`：三名玩家累计分牌分。
- `current_trick_cards`：当前牌墩已打出的牌。
- `trick_no`：已经开始的牌墩序号。
- `cumulative_bean_changes`：整局各玩家累计欢乐豆变化。
- `bean_balances`：最近一次成功结算后的欢乐豆余额。
- `player_triggered_boost`：影响终局排位的特殊牌型标记。

重连、命令重试或 Worker 换实例后，上述状态不得重置。

### 3.2 牌墩幂等流水

新增 `ddz_game_trick_settlement`：

- 唯一键：`(room_id, trick_no)`。
- 字段：`result_hash`、`requested_changes_json`、`actual_changes_json`、`balances_after_json`、`status`、`attempts`、`last_error`、创建/更新时间和完成时间。
- `result_hash` 只覆盖赢家、牌墩牌组、分值、底分、倍数和三方理论变化等不可变事实；实际扣款会受余额影响，不能参与重放时重新计算的摘要。

处理规则：

- 首次处理：锁定流水及真人档案，应用欢乐豆变化，标记完成。
- 相同键、相同摘要重放：返回首次保存的实际变化和结算后余额，不再次改余额。
- 相同键、不同摘要：抛出结算冲突，保留房间供人工或死信重放。
- 牌墩结算只更新欢乐豆，不增加 `total_games`、`wins`、段位星或战绩条数。

### 3.3 终局幂等流水

终局继续使用现有 `ddz_game_settlement` 和可靠结算队列：

- 结果摘要增加玩法、最终倍数、固定输赢、剩余手牌分变化和整局累计变化。
- 终局记录保存首次执行得到的实际终局变化与结算后余额，重放时直接复用，不根据已经变化的余额重新计算。
- 只应用终局固定输赢与剩余手牌分变化；已经成功入账的牌墩流水不得再次写入。
- `total_games`、`wins`、段位星和一人一条的 `GameRecord` 只在终局成功后更新一次。
- 终局任务失败时保留 Redis 房间，通过 RabbitMQ 重试/死信重放恢复；完成后再清理房间。

### 3.4 统一动作链路

单机真人、分布式真人、机器人、托管和超时动作必须调用同一个应用服务完成器：

1. 执行领域动作。
2. 完成牌墩或终局领域计算。
3. 幂等写入欢乐豆流水。
4. 保存 Redis 房间快照。
5. 生成 WebSocket/Outbox 事件。
6. 需要时安排下一回合超时任务。

禁止通过检查 Python 调用栈向事件列表注入 `trick_settled`；所有事件必须由显式结果字段转换。

## 5. 510K 专属 AI

### 5.1 现成模型结论

截至 2026-07-10，未检索到包含明确模型卡、训练规则、权重和可复现实验的 510K 预训练模型。

- [DouZero](https://github.com/kwai/DouZero) 面向标准斗地主，固定地主、地主上家、地主下家三个非对称角色及斗地主收益目标。
- [RLCard](https://github.com/datamllab/rlcard) 提供斗地主环境、规则模型和 DMC/DQN/NFSP/CFR 等训练框架，但不包含 510K 环境或预训练权重。

因此不能把当前 DouZero 权重直接用于 510K：初始手牌数量、角色结构、特殊动作空间、140 分牌池和欢乐豆奖励均不一致。可复用的是网络管理、候选动作评分和自博弈训练方法，不是现有权重。

### 5.2 当前里程碑：专属策略 AI

- 生成当前状态下全部合法动作，包括真/假 510K。
- 先保证合法性，再按以下因素排序：直接跑完、抢到高分牌墩、对手剩余牌数、牌权价值、拆牌成本、保留真510K/炸弹价值、倍数风险。
- 对关键候选执行有限次蒙特卡洛未知手牌采样，估计最终欢乐豆收益。
- 机器人、提示、真人托管共用排序结果；模型或采样异常时退回确定性合法动作。
- WebSocket 返回 `source: "fifty_k_strategy"`，不得误报为 `douzero`。

### 5.3 后续里程碑：自博弈模型

模型训练作为独立项目实施：

- 建立三人对称的 510K 环境和合法动作编码。
- 状态包含手牌、历史动作、剩余牌数、当前牌墩、累计分、底分和倍数。
- 奖励直接使用整局实际欢乐豆净变化，覆盖牌墩、固定 4500 和剩余分牌结算。
- 复用 DouZero/Deep Monte-Carlo 的自博弈思想，从头训练一个三席共享策略，而不是微调三角色斗地主权重。
- 只有模型在固定种子评测中稳定战胜策略 AI，且非法动作率为 0，才允许作为默认 AI；否则继续由策略 AI 托底。

## 6. 验收标准

- 前后端均满足 `王炸 > 真510K > 普通炸弹 > 假510K > 普通牌型`。
- 5/10/K 分值为 5/10/20，完整牌局的分牌归属可核对为 140 分。
- 牌墩和终局公式均乘牌局底分与对应倍数。
- 每名输家终局先扣 `4500 × 最终倍数`，再扣自己的剩余手牌分金额。
- 同一牌墩或终局任务重复执行不会重复扣豆。
- Redis 往返、重连和 Worker 切换后分数、牌墩、倍数和累计欢乐豆变化不丢失。
- 结算弹窗的三段明细之和等于 `total_bean_changes`，三名玩家实际变化总和符合资金来源规则。
- 510K 提示和托管始终返回合法动作，经典模式继续使用现有 DouZero。
- 实现完成后同步更新根目录 `README.md`；用户人工验证前不执行 Git 提交。
